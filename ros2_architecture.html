1.3.1 Node Structure

What is a Node?

A Node is the fundamental building block in ROS2. Think of it as a modular process that performs a specific computation or task within your robotic system. Each node is an independent entity that can communicate with other nodes.

Key Concept: Nodes are like individual workers in a factory. Each worker (node) has a specific job, and they communicate with each other to accomplish the overall goal of the system.

1.3.2 Graph Concepts

The ROS2 Computation Graph

The ROS2 graph is a peer-to-peer network of nodes that process data together. The graph represents how nodes, topics, services, and actions are connected and communicate with each other.

Graph Elements:

üìã

Topics

Named buses for streaming data. Publishers send messages, subscribers receive them. Many-to-many communication.

üîÑ

Services

Synchronous request-response pattern. Client sends a request, server processes and returns a response.

‚ö°

Actions

Asynchronous tasks with feedback. Like services but for long-running operations with progress updates.

üìä

Parameters

Configuration values associated with nodes. Can be changed at runtime without restarting nodes.

Real-World Example: Imagine a self-driving car where the camera node publishes images to a topic, the vision processing node subscribes to process them, and a planning node uses a service to calculate routes. All these nodes form the computation graph!

Graph Discovery

ROS2 uses DDS (Data Distribution Service) for automatic discovery. Nodes can find each other without a central master, making the system more robust and scalable.

Communication Type Pattern Best For

Services Request-Response Occasional computations (inverse kinematics, path planning)
Actions Goal-Feedback-Result Long-running tasks (navigation, manipulation)

1.3.3 Quality of Service (QoS)

What is QoS?

Quality of Service (QoS) policies in ROS2 define how data is transmitted between nodes. These policies ensure reliable communication in different network conditions and use cases.

Why QoS Matters: Different scenarios require different communication guarantees. A sensor streaming data can tolerate some packet loss, but a safety-critical command must be delivered reliably. QoS lets you configure this behavior!

Key QoS Policies:

üéØ

Reliability

BEST_EFFORT: Faster, may lose messages
RELIABLE: Guarantees delivery, slower

üì¶

History

KEEP_LAST: Store last N samples
KEEP_ALL: Store all samples (resource intensive)

‚è±Ô∏è

Durability

VOLATILE: Only for active subscribers
TRANSIENT_LOCAL: Late joiners get last message

‚è∞

Deadline

Maximum time between messages. Alerts if deadline is missed for time-critical data.

üîÑ

Liveliness

Mechanism to detect if a publisher is still active and publishing data regularly.

‚è≥

Lifespan

How long a message remains valid. Old messages are automatically discarded.

QoS Profiles

ROS2 provides predefined QoS profiles for common scenarios:

Profile Use Case Characteristics
Sensor Data High-frequency sensor streams Best effort, volatile, keep last 10
Parameters Configuration data Reliable, transient local, keep last 1000
Services Request-response Reliable, volatile, keep last 10
System Default General purpose Reliable, volatile, keep last 10
# Example: Setting QoS for a publisher from rclpy.qos import QoSProfile, ReliabilityPolicy, HistoryPolicy qos_profile = QoSProfile( reliability=ReliabilityPolicy.RELIABLE, history=HistoryPolicy.KEEP_LAST, depth=10 ) publisher = node.create_publisher( String, 'topic_name', qos_profile )
üí° Pro Tip: Publishers and subscribers must have compatible QoS settings to communicate. If a publisher uses BEST_EFFORT but subscriber requires RELIABLE, they won't connect!

An Executor is responsible for managing the execution of callbacks in ROS2. It handles when and how your node responds to incoming messages, service requests, and timer events.

Think of it this way: An executor is like a manager at a call center, deciding which incoming calls (callbacks) to answer and in what order. It ensures your node responds to events efficiently.

Types of Executors:

üîÑ

SingleThreaded

Executes all callbacks on one thread, sequentially. Simple and safe from race conditions.

‚ö°

MultiThreaded

Uses multiple threads to execute callbacks in parallel. Better performance but requires thread safety.

üéØ

StaticSingleThreaded

Optimized single-threaded executor with lower overhead for embedded systems.

What are Callbacks?

Callbacks are functions that get executed when certain events occur. They are the "responders" in your ROS2 system.

Callback Type Triggered By Common Use
Subscription Callback Receiving a message on a topic Processing sensor data, reacting to events
Timer Callback Periodic time intervals Publishing at fixed rates, periodic checks
Service Callback Receiving a service request Performing computations on demand
Action Callback Goal, feedback, or result events Long-running tasks with progress updates
# Example: Creating callbacks in ROS2 class MyNode(Node): def __init__(self): super().__init__('my_node') # Subscription callback - triggered by incoming messages self.subscription = self.create_subscription( String, 'topic', self.listener_callback, 10) # Timer callback - triggered every 1 second self.timer = self.create_timer(1.0, self.timer_callback) def listener_callback(self, msg): self.get_logger().info(f'Received: {msg.data}') def timer_callback(self): self.get_logger().info('Timer triggered!')

Callback Groups

Callback groups allow you to organize callbacks and control their execution behavior:

  • üîí MutuallyExclusive: Callbacks in this group cannot run simultaneously (default, thread-safe)
  • ‚ö° Reentrant: Callbacks can run in parallel (requires thread-safe code, better performance)
# Example: Using callback groups with MultiThreadedExecutor from rclpy.callback_groups import ReentrantCallbackGroup from rclpy.executors import MultiThreadedExecutor callback_group = ReentrantCallbackGroup() timer = node.create_timer( 1.0, timer_callback, callback_group=callback_group ) executor = MultiThreadedExecutor() executor.add_node(node) executor.spin()
‚ö†Ô∏è Important: When using MultiThreadedExecutor with ReentrantCallbackGroup, ensure your callbacks are thread-safe! Use locks or other synchronization mechanisms to protect shared data.

Spin Functions

Spin functions keep your node alive and processing callbacks:

Function Behavior Use Case
spin() Blocks and processes callbacks indefinitely Standard node operation
spin_once() Processes callbacks for a specified timeout Integrating with other event loops
spin_until_future_complete() Spins until a specific future completes Waiting for service responses or actions
